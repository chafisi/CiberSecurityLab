1. Fundamentos: Teoría de la Información

Introducida por Claude Shannon (1948), la codificación de la fuente busca representar mensajes de forma eficiente eliminando redundancia sin perder información esencial.

📊 Entropía (H)

Mide la incertidumbre media de una fuente aleatoria. Es el límite teórico de compresión sin pérdida.

H(X) = − Σ p(xi) · log₂ p(xi)

* Alta entropía = Aleatorio = Difícil de comprimir.
* Baja entropía = Predecible = Fácil de comprimir.

♻️ Redundancia

Información repetitiva o predecible. Los algoritmos de compresión eliminan esto para reducir el tamaño (tasa de compresión r).

Original ➡️ Comprimido

⚠️ Importancia en Seguridad

  • Optimización: Reduce volumen para cifrar (menos CPU/ancho de banda).
  • Orden Crítico: Comprimir ➔ Cifrar es recomendado. Cifrar ➔ Comprimir es ineficaz (cifrado produce alta entropía/incompresibilidad).
  • Riesgo: Si se co-comprimen secretos con datos del usuario, se crean oráculos de longitud (CRIME/BREACH).

2. Técnicas de Codificación de la Fuente

🌳 Huffman

Código Prefijo Óptimo

Asigna códigos más cortos a símbolos frecuentes. Ideal para fuentes estáticas.

  • ✅ Rápido y simple.
  • ❌ Granularidad de 1 bit (no supera H+1).
  • 🛡️ Usado en DEFLATE (ZIP, PNG), HPACK.
🔢 Aritmética

Intervalos

Codifica secuencias completas en un subintervalo [0,1). Muy preciso.

  • ✅ Se acerca mucho a la entropía H.
  • ❌ Complejidad computacional alta.
  • 🛡️ Usado en códecs modernos (JPEG 2000, CABAC).
🔄 Lempel-Ziv

Universal Adaptativo

Aprende patrones "al vuelo" usando diccionarios o ventanas deslizantes.

  • ✅ No requiere conocer la distribución previa.
  • ❌ Riesgo de co-compresión (CRIME).
  • 🛡️ Base de ZIP, GZIP, Brotli, Zstd.

Comparativa Técnica

Técnica Fortalezas Limitaciones Usos Típicos
Huffman Rápido, óptimo para fuentes fijas Pierde eficacia si cambia distribución DEFLATE, PNG, HPACK
Aritmética Precisión extrema (~H) Complejidad numérica Códecs de video/audio
LZ (77/78) Universal, aprende al vuelo Vulnerable a CRIME/BREACH ZIP, GZIP, Brotli, Web

3. Relación con Criptografía y Canales Laterales

Regla de Oro: Comprimir ANTES de Cifrar

El orden correcto (Compresión ➔ Cifrado) reduce el volumen y elimina patrones estadísticos antes de protegerlos.

Datos Originales ➔ [Compresión] ➔ Datos Comprimidos ➔ [Cifrado AEAD] ➔ Tráfico Seguro

❌ El Peligro: Co-Compresión

Si se comprimen JUNTOS:
1. Secretos (Cookies/Tokens)
2. Datos controlados por el atacante

El tamaño final revela coincidencias (CRIME/BREACH).

Buenas Prácticas Defensivas

  • Desactivar compresión TLS: Usar compresión solo en capa de aplicación si es necesario.
  • Separar contextos: Nunca mezclar cookies con inputs del usuario en la misma respuesta comprimida.
  • Padding Aleatorio: Añadir bits "basura" para ocultar la longitud real del mensaje.
  • Autenticación (AEAD): Usar AES-GCM o ChaCha20-Poly1305 siempre.

Riesgos Comunes

  • Zip-Bombs: Archivos pequeños que explotan memoria/CPU al descomprimir (DoS).
  • Oráculos de Longitud: Variación del ciphertext que permite deducir secretos.
  • Metadatos Exif: En imágenes, pueden filtrar ubicación o dispositivo.

4. Caso Práctico 16: CRIME en TLS/HTTP

Análisis de filtración por compresión de cabeceras (HPACK/QPACK) y soluciones.

🕵️‍♂️

Enunciado: Ataque CRIME/BREACH

Web Security

El ataque CRIME explota la compresión a nivel TLS o SPDY para inferir secretos (cookies). Al inyectar cadenas candidatas y observar si el tamaño del ciphertext disminuye (mayor compresión), se descubren partes de la cookie. Similarmente, BREACH lo hace a nivel de cuerpo HTTP. Relevante en HTTP/2–3 donde HPACK/QPACK pueden crear canales laterales si no se gestionan contextos.

MIS RESPUESTAS SOLUCIÓN

1. Clasificar requisitos para evitar filtraciones
🟢 Funcionales
  • Separar datos secretos (cookies) de entradas del usuario.
  • Desactivar compresión TLS si no es vital.
  • Gestionar contextos HPACK/QPACK evitando mezcla.
🔵 No funcionales
  • Mantener rendimiento aceptable sin compresión TLS.
  • Minimizar latencia por padding.
🟣 Técnicos
  • Desactivar compresión TLS (obligatorio en 1.3).
  • Aplicar padding aleatorio y rate limiting.
  • Sanitizar cabeceras sensibles.
2. Elaborar caso de uso: Evitar co-compresión

Caso de uso: Protección contra filtración por compresión

Actor: Aplicación web / Servidor HTTP.

Flujo principal:

  1. Cliente envía petición con parámetros controlados.
  2. Servidor genera respuesta con cookies/tokens.
  3. Acción Crítica: Sistema evita co-comprimir datos sensibles:
    • Se envían en cabeceras NO comprimidas.
    • O se separan contextos en HPACK/QPACK.
    • O se añade padding forzado.
  4. Comunicación se cifra en TLS 1.3 (sin compresión).
  5. Usuario recibe respuesta sin revelar info por longitud.

Excepciones:

  • Uso de HTTP/2/3 sin políticas seguras → Activar separación estricta.
  • Necesidad de ancho de banda → Padding aleatorio obligatorio.
3. Métodos de validación (Oráculos de longitud)
  • Entorno especializado: Comprobar variación predictiva del ciphertext al modificar inputs.
  • Escaneos automáticos: Herramientas que detecten compresión indebida o riesgo HPACK.
  • Análisis de tráfico: Verificar que TLS no usa compresión y no hay correlación input/tamaño.
  • Revisión de políticas: Confirmar aislamiento entre cabeceras sensibles y userInput.

5. Caso Práctico 15: Huffman en JPEG y transmisión segura

Análisis del pipeline "comprimir -> cifrar" en imágenes y canales laterales.

🖼️

Enunciado: JPEG & Canal Lateral

Multimedia

Plataformas convierten imágenes a JPEG (DCT + Cuantización + Huffman). Esto reduce tamaño y acelera subida. Aunque se cifran con TLS 1.3, la longitud del ciphertext puede revelar resolución, calidad o tipo de imagen (canal lateral). En contextos sensibles (médico/legal), esto es crítico.

MIS RESPUESTAS SOLUCIÓN

1. Clasificar requisitos para pipeline seguro
🟢 Funcionales
  • Comprimir antes de cifrar (para eficiencia).
  • Transmitir vía canal cifrado (TLS 1.3).
  • Aplicar padding opcional si tamaño revela info.
🔵 No funcionales
  • Minimizar latencia de compresión.
  • Mantener calidad perceptual.
  • Evitar patrones de longitud correlacionables.
🟣 Técnicos
  • Estándar JPEG con Huffman.
  • TLS 1.3 con AES-GCM/ChaCha20.
  • Padding uniforme o tamaños normalizados.
  • Eliminar metadatos EXIF.
2. Elaborar caso de uso: Transmisión segura JPEG

Caso de uso: Subida segura de imagen

Actor: Cliente y Servidor.

Flujo principal:

  1. Usuario sube imagen.
  2. App convierte a JPEG (Huffman).
  3. Comunicación se cifra (TLS 1.3).
  4. Servidor recibe y procesa.
  5. Sistema registra tamaño interno.

Excepciones:

  • Imágenes con metadatos sensibles → Normalización previa (limpieza EXIF).
  • Tamaño revela info → Aplicar padding uniforme antes de cifrar.
3. Métodos de validación (No filtración por longitud)
  • Pruebas canal lateral: Comparar varios archivos similares; verificar si ciphertext length infiere resolución/tipo.
  • Evaluación rendimiento: Ahorro de ancho de banda vs. PNG/RAW.
  • Verificación TLS: Usar SSL Labs/Wireshark para confirmar cifrado robusto y sin fallback.
  • Análisis privacidad: Chequear eliminación de EXIF y pruebas de padding.

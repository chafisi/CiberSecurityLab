🛡️ Respuestas Tipo: Criptografía y Seguridad

Usa esta plantilla como referencia rápida para cualquier pregunta sobre criptografía aplicada.

Puntos Clave (Palabras Mágicas)

  • Diferenciación: Simétrico (misma clave) vs Asimétrico (clave pública/privada).
  • Integridad: Funciones hash (SHA-256, SHA-3) siempre mencionadas junto a firma.
  • Gestión Claves: Rotación, almacenamiento seguro (HSM/KMS), vida útil limitada.
  • No Repudio: Solo asimétrico proporciona no repudio legal.

Tecnologías Criptográficas

  • Simétrico: AES-256, ChaCha20, 3DES (obsoleto), Blowfish.
  • Asimétrico: RSA-4096, ECDSA (P-256/P-384), Ed25519.
  • Hash: SHA-256, SHA-384, SHA-512, BLAKE3.
  • Infraestructura: PKI, X.509, OCSP, CRL, HSM, TLS 1.3.

Nivel de Fuerza Criptográfica Recomendada

Nivel Seguridad Simétrico Asimétrico Use Case
Bajo AES-128 RSA-2048 / ECC-256 Datos no críticos
Medio AES-256 RSA-3072 / ECC-384 Transacciones financieras
Alto AES-256-GCM RSA-4096 / ECC-521 Gobierno, Salud, Nuclear

Errores Comunes a Evitar en Exámenes

  1. Nunca usar DES/MD5: Están rotos criptográficamente desde hace años.
  2. Claves cortas = inseguridad: Mínimo AES-128, RSA-2048 (recomendado 256/4096).
  3. IV debe ser único: Reutilizar IV en modo CBC/CTR compromete seguridad.
  4. Firma ≠ Cifrado: Firma es hash + clave privada, cifrado es dato + clave pública.
  5. Siempre mencionar normativas: eIDAS, FIPS 140-2, RGPD Art. 32 dan contexto legal.

🏛️ Respuestas Tipo: PKI y Firma Digital

Guía rápida para preguntas sobre infraestructura de confianza y validez legal.

Puntos Clave (Palabras Mágicas)

  • Cadena de Confianza: Raíz (Trust Anchor) → Intermedia → Final (Usuario/Servidor).
  • Revocación: Es vital mencionar CRL (Listas) y OCSP (Online) como mecanismos complementarios.
  • Tipo de Firmas eIDAS: Simple (baja seguridad), Avanzada (F.A.), Cualificada (F.C., igual a manuscrita).
  • Sello de Tiempo: Necesario para probar que la firma existía en un momento X (anti-replay y validez futura).

Componentes Técnicos Clave

  • HSM: Hardware Security Module, obligatorio para proteger la clave de la CA Raíz.
  • X.509 v3: Formato estándar de certificado (Subject, Issuer, Serial, Validity, Extensions/KU/EKU).
  • EKU (Extended Key Usage): Define uso (Servidor Web, Cliente, Código, Email).
  • OID: Identificadores únicos para algoritmos y tipos de certificación.

Niveles de Firma según eIDAS

Nivel Identificación Características Ejemplo
Simple Débil o baja Sin garantía alta de origen Firma scaneada, checkbox
Avanzada (FEA) Única identificación Control exclusivo firma, vinculado datos Certificado cualificado no presencial
Cualificada (FEC) Muy Alta Dispositivo cualificado (token/biométrico) DNIe3+ / Cl@ve PIN con Token

Preguntas Frecuentes en Exámenes

  1. "¿Quién emite el certificado?": Una Autoridad de Certificación (CA) confiable.
  2. "¿Qué pasa si se filtra la clave privada?": Debe revocarse inmediatamente mediante OCSP/CRL.
  3. "¿Es lo mismo cifrar que firmar?": No. Cifrar usa clave pública del receptor (confidencialidad). Firmar usa clave privada del remitente (integridad/no repudio).
  4. "¿Por qué la CA Raíz suele estar offline?": Para minimizar el riesgo de compromiso. Solo enciende para emitir intermedias.
  5. "¿Qué es el 'Chain of Trust'?": La validación recursiva desde el certificado final hasta la raíz de confianza.

🛡️ Respuestas Tipo: Cortafuegos y NAT

Guía rápida para preguntas sobre protección perimetral y gestión de tráfico.

Puntos Clave (Palabras Mágicas)

  • Principio Mínimo Privilegio: "Deny All" por defecto, solo permitir lo explícitamente necesario.
  • Zonas de Seguridad: LAN (Trusted), WAN (Untrusted), DMZ (Semi-trusted). Niveles de confianza distintos.
  • Inspección de Estado (Stateful): El firewall recuerda el contexto de la conexión (no solo paquetes sueltos).
  • Diferencia NAT vs Firewall: NAT oculta/trasladan IPs; Firewall decide permitir/denegar tráfico basado en reglas.

Tecnologías y Conceptos

  • NAT Tipos: SNAT (Source - Saliente), DNAT (Destino - Entrante/Port Forwarding), Masquerading (Dinámico).
  • NGFW (Next Gen): Capa 7 (App-aware), DLP integrado, IPS/IDS, Identidad de usuario (User-ID).
  • Hairpinning: Acceder al servicio interno desde dentro usando la IP pública (loopback NAT).
  • High Availability (HA): Clusters Activo-Activo o Activo-Pasivo para redundancia.

Tipos de Cortafuegos Evolutivos

Tipo Capas OSI Característica Ventaja/Desventaja
Estático Red/Transporte (3-4) Filtrado por IP/Puerto simple + Rápido / - Poco seguro
Stateful Red/Transporte (3-4) Memoria de estado de conexión + Más seguro / - Mayor CPU
Proxy/App Aplicación (7) Intermediario, entiende protocolo + Muy seguro / - Latencia alta
NGFW 2-7 (Todo) IPS, App Control, User-ID + Completo / - Costoso

Errores Comunes en Configuración

  1. "Any/Any" permisivo: NUNCA poner una regla que permita todo sin motivo específico. Rompe el principio de menor privilegio.
  2. Orden de reglas incorrecto: Las reglas se evalúan de arriba a abajo. Si ponas "Deny Todo" arriba, las permitidas abajo nunca se usan.
  3. NAT sin Firewall: La NAT no es un método de seguridad por sí mismo (no es un firewall). Siempre debe combinarse con reglas de acceso.
  4. Olvidar la regla de retorno: En firewalls stateful suele ser automático, pero en filtros estáticos debes permitir el tráfico de respuesta explícitamente.
  5. No auditar logs: Un firewall sin logging es ciego ante intentos de intrusión o malas configuraciones.

2. Flujo de Detección por Firma

Actor

Sistema IDS: Intercepts el paquete HTTP entrante y lo reensambla en el buffer de memoria.

Acción:** Alertar al SOC (SIEM) y bloquear la conexión TCP inmediatamente (Modo IPS).

5️⃣ Registro

Log Forense: Se guarda el PCAP del ataque para análisis posterior y evidencia.

💡 Clave: La eficacia depende totalmente de que la firma exista en la base de datos. No detecta ataques zero-day.

👁️ Respuestas Tipo: IDS e IPS

Guía rápida para diferenciar tipos de detección y estrategias de respuesta.

Conceptos Clave (Palabras Mágicas)

  • IDS vs IPS: IDS = Monitorizar y Alertar (Pasivo). IPS = Monitorizar y Bloquear (Activo).
  • Firma (Signature): Detecta lo conocido. Exacto, pero inútil contra nuevos ataques.
  • Anomalía (Behavior): Detecta lo extraño. Bueno para Zero-day, pero propenso a falsos positivos.
  • False Positive/Negative: El equilibrio perfecto es difícil. Un FP alerta de algo inofensivo; un FN deja pasar un ataque real.

Tecnologías y Herramientas

  • Snort / Suricata: Los estándares open-source. Basados en reglas, muy configurables.
  • Zeek (antes Bro): Enfocado en análisis de protocolo y logs detallados, no solo alertas binarias.
  • Darktrace / CrowdStrike: Soluciones comerciales con IA avanzada para detección de anomalías.
  • Honeypot: Trampa deliberada para atraer atacantes y estudiar su comportamiento (tipo especial de detección).

Preguntas Clave de Examen

  1. "¿Cómo funciona un IDS?": Escucha pasivamente (SPAN port/tap), analiza paquetes, compara con reglas o patrones, genera alertas.
  2. "¿Cuál es la diferencia entre HIDS y NIDS?": HIDS (Host) vigila archivos/logs del SO local. NIDS (Network) vigila el tráfico de la red.
  3. "¿Qué es un False Negative?": Cuando el sistema NO detecta un ataque real. El peor escenario posible.
  4. "¿Por qué necesitamos ambos enfoques?": Las firmas dan precisión inmediata; las anomalías nos salvan de lo desconocido. La defensa en profundidad los combina.
  5. "¿Qué pasa si el atacante cifra todo?": El IDS/IPS pierde eficacia si no puede hacer SSL Decryption (Man-in-the-Middle en el firewall). Necesitamos ver el contenido.

Caso Práctico 11

Implementación de Túneles IPsec para Conexiones Site-to-Site

Establecimiento de conexión segura entre dos sedes corporativas utilizando protocolo IPsec en modo Túnel, garantizando confidencialidad e integridad del tráfico inter-sede.

1. Matriz de Requisitos

Categoría Detalles (Verbos/Métricas/Tecnologías)
🔹 Funcionales Negociar fase 1 (IKE) y fase 2 (ESP/AH), Autenticar peer mediante PSK o Certificados, Cifrar tráfico de red completa, Verificar integridad de paquetes.
📊 No Funcionales Throughput del túnel > 500 Mbps, Overhead de cifrado ≤ 5%, Tiempo reconexión tras fallo < 30s, MTU ajustado para evitar fragmentación.
⚙️ Técnicos IKEv2 (RFC 7296), ESP con AES-256-GCM, SHA-2 para integridad, DH Grupo 14/19, Perfect Forward Secrecy (PFS), NAT-T para detrás de firewalls.
⚖️ Legales RGPD (Tráfico transfronterizo), ENS (Comunicaciones seguras), Ley de protección de infraestructuras críticas, Normativa de exportación criptográfica.

2. Flujo de Establecimiento del Túnel IPsec

Actor

Sede A ↔ Sede B: Dos routers/firewalls corporativos que necesitan conectar redes privadas (192.168.1.0/24 ↔ 10.0.0.0/24).

1️⃣ Fase 1 - IKE SA

Negociación: Los peers intercambian mensajes IKE para autenticarse (PSK/Certificado) y acordar algoritmos de cifrado.

2️⃣ Fase 1 - ISAKMP SA

Canal Seguro: Se establece un canal cifrado protegido para negociar los parámetros de IPsec (la "tubería de gestión").

3️⃣ Fase 2 - Quick Mode

Acuerdo IPsec: Negocian los parámetros del túnel real: qué tráfico proteger (proxy IDs), qué cifrar (AES-256), cuánto tiempo (lifetime).

4️⃣ Fase 2 - SA Creada

Túnel Activo:** Se crean las Security Associations (SA) bidireccionales. Todo el tráfico entre subnets ahora se encapsula y cifra.

5️⃣ Keepalive & Rekey

Mantenimiento: Dead Peer Detection (DPD) verifica conectividad periódica. PFS renueva claves antes de expirar sin exponer datos pasados.

💡 Clave: El Modo Túnel encapsula TODO el paquete IP original (cabecera + datos), ideal para site-to-site. El Modo Transporte solo cifra payload (host-to-host).

Protección contra Ataques Denegación de Servicio (DoS/DDoS)

Implementación de mecanismos de mitigación de ataques DDoS a nivel de red (Capa 3/4) contra servidores web públicos, usando rate limiting y sinks de limpieza.

1. Matriz de Requisitos

Categoría Detalles (Verbos/Métricas/Tecnologías)
🔹 Funcionales Detectar picos de tráfico anormales, Aplicar rate limiting por IP, Filtrar SYN Flood, Challenge HTTP (CAPTCHA) para bots, Redirigir tráfico limpio (Scrubbing Center).
📊 No Funcionales Tiempo detección ataque < 30s, Capacidad mitigación ≥ 1 Tbps, Falsos positivos ≤ 0.1%, Disponibilidad servicio durante ataque ≥ 99.5%.

⚖️ Legales Ley de Servicios de la Sociedad de la Información (responsabilidad intermedios), RGPD (si afecta datos personales), NIS2 (protección operadores esenciales), SLA contractual.

2. Flujo de Mitigación DDoS

Actor

Botnet Atacante: 10k dispositivos IoT lanzan un ataque volumétrico SYN Flood hacia el servidor web objetivo (IP Pública).

1️⃣ Detección

Sistema IDS/SCRUBBING: Detecta anomalía súbita: 50 Gbps de tráfico entrante (vs 10 Mbps normales). Umbral superado.

2️⃣ Alarma

Alerta Automática: Se activa procedimiento de emergencia DDoS. Notificación al equipo SOC y proveedor de mitigación.

3️⃣ Redirección

BGP Update / DNS: Se desvía todo el tráfico de la IP atacada a través del centro de limpieza (Scrubbing Center) del proveedor CDN.

4️⃣ Limpieza

Filtrado Inteligente: El Scrubbing analiza paquetes: identifica síntesis maliciosos, aplica SYN Cookies, bloquea IPs sospechosas, deja pasar tráfico legítimo.

5️⃣ Retorno

Tráfico Limpio: Solo el tráfico validado llega a tu servidor. El ataque es absorbido y disipado en la red del CDN. Usuario final no nota impacto.

⚠️ Clave: La protección DDoS debe estar fuera de tu red (on-cloud/on-prem híbrido). Si atacas desde dentro, el atacante satura tu enlace ANTES de llegar al filtro.

Guía rápida para protocolos de capa de red, VPNs y protección de disponibilidad.

Puntos Clave (Palabras Mágicas)

Protocolos y Tecnologías

  • VPN Tipos: Site-to-Site (IPsec), Remote Access (OpenVPN, WireGuard, SSL-VPN), MPLS L2/L3 VPN.

  • Anycast: Mismo IP en múltiples lugares. Mejora rendimiento y mitiga DDoS (trafgo se dispersa geográficamente).
  • Honeynets: Redes cebada para estudiar tráfico malicioso sin riesgo a producción.

Preguntas Frecuentes de Examen

  1. "¿Por qué IPsec necesita IKE?": Para negociar de forma segura los parámetros de cifrado y autenticar los peers antes de enviar datos sensibles.
  2. "¿Qué es Perfect Forward Secrecy (PFS)?": Garantiza que si se compromete una clave privada a largo plazo, las sesiones pasadas no puedan ser descifradas (nuevas claves por sesión).
  3. "¿La NAT rompe IPsec?": Sí, porque IPsec autentifica la cabecera IP original (incluyendo direcciones). Por eso necesitamos NAT-T (UDP encapsulation en puerto 4500).
  4. "¿Cuál es la diferencia entre ataque DoS y DDoS?": DoS = Denegación de Servicio (origen único). DDoS = Distribuido (múltiples fuentes coordinadas, ej: botnet). Más difícil de mitigar.
  5. "¿Por qué usar Anycast para DDoS?": Porque dispersa el tráfico atacante geográficamente. El atacante no puede saturar un punto único de entrada; su capacidad se reparte.

Prevención de Inyección SQL (SQLi)

1. Matriz de Requisitos

Categoría Detalles (Verbos/Métricas/Tecnologías)
🔹 Funcionales Validar entrada del usuario, Sanitizar parámetros, Ejecutar consultas preparadas (Prepared Statements), Regresar solo datos autorizados, Registrar intentos de inyección.

⚖️ Legales RGPD Art. 32 (Seguridad tratamiento), Ley de Delitos Informáticos (acceso ilícito), GDPR breach notification si hay fuga datos, ISO 27001 (A.14 Desarrollo seguro).

2. Flujo de Ataque y Prevención SQLi

Actor

Intrusista: Introduce `' OR '1'='1` en el campo username para intentar acceder sin contraseña.

1️⃣ Entrada Vulnerable

Código Malo: `SELECT * FROM users WHERE user = '` + input + `' AND pass = '` + pass + `'` (Concatenación directa). ¡Peligro!

2️⃣ Payload Injectado

Resultado SQL: `SELECT * FROM users WHERE user = '' OR '1'='1' AND pass = '...'`. La condición siempre es verdadera.

3️⃣ Explotación

Consecuencia: Base de datos devuelve el primer usuario (admin). El atacante accede sin contraseña conocida.

4️⃣ Prevención Correcta

Solución: Usar Prepared Statements: `?` placeholders. La DB interpreta el input SOLO como DATOS, no como CÓDIGO ejecutable.

5️⃣ Defensa Profundidad

Capas Extra: WAF detecta patrones SQL maliciosos antes de llegar a app. DB con permisos mínimo necesarios (solo SELECT en tabla users).

✅ Clave: Las consultas parametrizadas separan CÓDIGO de DATOS. Es la medida más efectiva contra SQLi. El WAF añade una segunda línea de defensa.

2. Flujo de Protección XSS y CSRF

Actor

2️⃣ Escenario CSRF

3️⃣ Mitigación XSS

5️⃣ Verificación Origin

Extra: Server checkea headers `Origin` y `Referer`. Si vienen de dominio diferente al esperado, bloquear request inmediatamente.

💡 Clave: XSS compromete CONFIDENCIALIDAD (roba datos cliente). CSRF compromete INTEGRIDAD (realiza acciones no deseadas). Ambos requieren controles diferentes pero complementarios.

Puntos Clave (Palabras Mágicas)

Preguntas Frecuentes de Examen

  1. "¿Por qué validar INPUT no basta?": Porque datos pueden ser modificados en tránsito o venir de fuentes no controladas. Siempre validar TAMBIÉN en servidor.
  2. "¿Qué es SameSite en cookies?": Atributo que limita cuándo el navegador envía cookies (Strict/Lax/None). Previene ataques CSRF transmitiendo credenciales entre sitios.
  3. "¿SQL Blind vs. Error-based Injection?": Error-based muestra mensajes DB visibles. Blind no muestra errores, pero inferimos datos mediante respuestas True/False o time delays.
  4. "¿CSP puede bloquear todo el JS?": Sí, con política ultra-restrictiva. Pero normalmente permitimos scripts propios ('self') y nonce/hash específicos para funcionalidad necesaria.
  5. "¿Por qué HttpOnly en cookies?": Para que JavaScript NO pueda leer la cookie. Evita robo de sesión vía XSS, aunque el atacante aún pueda USARla si ya está en navegador víctima.

Caso Práctico 15

Compresión Sin Pérdida para Datos Críticos

Implementación de algoritmo Huffman para comprimir archivos de log y bases de datos SQL, garantizando que cada bit original pueda ser recuperado exactamente tras la descompresión.

1. Matriz de Requisitos

Categoría Detalles (Verbos/Métricas/Tecnologías)
🔹 Funcionales Analizar frecuencia de símbolos, Construir árbol binario óptimo, Asignar códigos prefijo-free, Empaquetar flujo de bits, Reconstruir origen exacto al descomprimir.
📊 No Funcionales Ratio compresión ≥ 40% (texto repetitivo), Entropía maximizada, Tiempo codificación/decodificación simétrico, Overhead CPU bajo (≤ 15%).
⚙️ Técnicos Algoritmo Huffman (estático/adaptativo), LZW (Lempel-Ziv-Welch), Deflate (ZIP/GZIP), Aritmética Coding, Diccionarios dinámicos.
⚖️ Legales Normativa de retención de datos (logs comprados deben ser legibles por ley), estándares ISO/IEC 11172 (MPEG) parte de compresión, Licencias de software (patentes LZW históricas).

2. Flujo de Compresión Huffman

Actor

Sistema de Archivos: Necesita guardar un archivo de texto gigante "DATA.LOG" ocupando menos espacio sin perder ni un solo carácter.

1️⃣ Análisis de Frecuencia

Escaneo: El algoritmo cuenta cada símbolo. Ej: 'E' aparece 50 veces, 'Z' solo 1 vez, 'A' 20 veces.

2️⃣ Construcción del Árbol

Lógica: Se crea un árbol binario. Los símbolos más frecuentes están más cerca de la raíz (códigos cortos), los raros en las hojas profundas (códigos largos).

3️⃣ Asignación de Códigos

Mapeo: Caminando del padre a hijo izquierdo = '0', derecho = '1'. Resultado: 'E' → `0`, 'Z' → `11101`. ¡Ningún código es prefijo de otro!

4️⃣ Sustitución y Empaque

Compresión: Se reemplazan todos los caracteres originales por sus nuevas cadenas de bits. Se guarda el árbol (o tabla) al inicio del archivo.

5️⃣ Descompresión Inversa

Recuperación: Al leer, se usa el árbol guardado. Cada bit guía por el árbol hasta llegar a una hoja (carácter original). ¡Reconstrucción 100% exacta!

✅ Clave: La compresión sin pérdida funciona explotando la redundancia estadística. Si el archivo ya es aleatorio (entropía alta), no se puede comprimir mucho. Ideal para texto, código, JSON.

Compresión Con Pérdida para Multimedia Streaming

Optimización de archivos de audio y video para streaming web, utilizando técnicas de eliminación de información imperceptible al ojo humano (transformada DCT, sub-muestreo de crominancia).

1. Matriz de Requisitos

Categoría Detalles (Verbos/Métricas/Tecnologías)
🔹 Funcionales Transformar espacio a frecuencia (DCT), Cuantificar coeficientes (redondeo agresivo), Eliminar frecuencias inaudibles/invisibles, Codificar residuos, Generar stream continuo.
📊 No Funcionales Ratio compresión > 90% (ej. DVD a Web), Latencia de buffering < 2s, Calidad perceptiva (PSNR/SSIM) > 40dB, Compatibilidad universal (H.264/H.265, AAC).
⚙️ Técnicos Codecs: MP3 (Audio), H.264/AVC, H.265/HEVC, VP9, AV1. Técnicas: Transformada Cosinus Discreta (DCT), Sub-muestreo 4:2:0, Inter-frame prediction (P/B frames).
⚖️ Legales Propiedad intelectual de codecs (patentes MPEG-LA, Apple), DRM (Gestión de Derechos Digitales), Normativas de accesibilidad (subtítulos obligatorios en ciertos formatos).

2. Flujo de Compresión de Video (H.264)

Actor

Streaming Service: Debe transmitir un evento en vivo en 4K pero el ancho de banda del usuario es limitado (5 Mbps).

1️⃣ Transformación Espacial

DCT:** Se divide la imagen en bloques 8x8 píxeles. Se convierte de valores de color (RGB/YUV) a coeficientes de frecuencia (bajas = detalles grandes, altas = detalles finos/ruido).

2️⃣ Cuantización (La Pérdida)

Reducción: Se dividen los coeficientes por un valor grande y se redondean. Muchos coeficientes de alta frecuencia (detalles finos que el ojo apenas ve) se vuelven CERO. ¡Aquí se pierde info!

3️⃣ Predicción Temporal

Eficiencia: En lugar de enviar todo el cuadro (Frame), solo se envían las diferencias respecto al cuadro anterior (Motion Estimation). Solo se codifican objetos en movimiento.

4️⃣ Codificación Entropía

Compactación final: Se aplica CAVLC o CABAC a los ceros y unos resultantes. Los bloques llenos de ceros se comprimen muchísimo.

5️⃣ Stream Adaptativo

Resultado: Se genera un bitrate menor. El decodificador reconstruye la imagen usando los datos recibidos; la calidad parece buena aunque faltan datos invisibles.

⚠️ Clave: La pérdida es irreversible. Una vez cuantificado, ese dato no existe. El objetivo es que la pérdida sea imperceptible para el humano (psicovisual) o inaudible (psicoacústica).

📦 Respuestas Tipo: Compresión de Datos

Guía rápida para entender entropía, algoritmos y cuándo usar cada tipo de compresión.

Conceptos Clave (Palabras Mágicas)

  • Entropía: Medida de incertidumbre o información promedio. Máxima entropía = aleatoriedad pura (imposible comprimir). Minima entropía = mucha redundancia (fácil comprimir).
  • Diccionario vs. Estadístico: LZW/ZIP usan diccionarios (repitiendo patrones). Huffman/Aritmético usan estadísticas (símbolos frecuentes = bits menos). Combinados son lo mejor (Deflate).
  • Cuantización: Paso irreversibel donde se descarta información. Es el corazón de la compresión con pérdida.
  • Psico-acústica / Psico-visuales: Modelos que explotan limitaciones humanas (no oímos frecuencias muy agudas/bajas juntas, no vemos cambio de color tan rápido como brillo).

Formatos y Codecs Comunes

  • Sin Pérdida: ZIP, GZIP, PNG (imágenes simples), FLAC (audio), ALAC. Recuperación bit-a-bit.
  • Con Pérdida: JPEG (fotos), MP3 (audio), H.264/HEVC/AV1 (video). Uso masivo en internet.
  • Híbridos: Algunos formatos soportan ambos modos o niveles ajustables de pérdida.
  • Codecs Modernos: AV1 (open source, superior a H.265), Opus (audio de baja latencia).

Comparativa: Sin Pérdida vs. Con Pérdida

Característica Sin Pérdida (Lossless) Con Pérdida (Lossy)
Información Original Totalmente recuperable Parcialmente irrecuperable
Ratio Compresión Bajo/Medio (2:1 a 5:1) Alto (10:1 a 50:1+)
Tipo de Datos Texto, Código, DB, Archivos ejecutables Audio, Imágenes, Video
Complejidad Generalmente menor Mayor (requiere modelos complejos)

¿Por qué no usar siempre Lossless?

Si bien lossless es perfecto, la realidad del ancho de banda y almacenamiento nos obliga a usar lossy para multimedia. Imagina streaming Netflix en lossless: necesitarías 100 Mbps constantes por usuario. Con lossy inteligente (H.265), bastan 15-20 Mbps para 4K excelente.

Regla de oro: Usa Lossless si un solo bit cambiado rompe la función (código, finanzas, salud). Usa Lossy si la percepción humana importa más que el dato crudo (entretenimiento).

Preguntas Frecuentes de Examen

  1. "¿Qué es el código prefijo-free?": Que ningún código asignado a un símbolo comienza con otro código asignado. Esto permite decodificar el flujo de bits sin comas ni separadores, mirando bit a bit.
  2. "¿Por qué JPEG tiene artefactos cuadrados?": Porque trabaja con bloques de 8x8 píxeles (DCT). Si la compresión es muy agresiva, los bordes de estos bloques se notan (blocking artifacts).
  3. "¿Qué diferencia hay entre compresión espacial y temporal?": Espacial = comprime dentro de una sola imagen (redundancia de colores adyacentes). Temporal = comprime entre imágenes consecutivas (cosas que no se mueven).
  4. "¿El formato PNG es con o sin pérdida?": Sin pérdida. Para fotos suele usarse JPG (con pérdida) porque el ratio es mejor, pero PNG es ideal para gráficos con texto o líneas nítidas.
  5. "¿Qué es la entropía máxima?": Cuando todos los símbolos aparecen con igual probabilidad. En ese caso, cualquier intento de compresión estadística falla o incluso agranda el archivo.

Ejercicio 3: Insider con USB no autorizado

Insider / Malware
Enunciado:

En una institución pública, un empleado conectó un dispositivo USB no autorizado que contenía malware. Al ejecutarse, el código malicioso permitió la extracción de información sensible relacionada con proyectos estratégicos del organismo. La investigación reveló la inexistencia de políticas claras sobre el uso de dispositivos externos, la falta de controles DLP (Data Loss Prevention) y la ausencia de bloqueos físicos o lógicos en los puertos USB de los equipos. Este incidente puso de manifiesto el riesgo asociado a las amenazas internas, tanto accidentales como maliciosas, cuando no existen medidas de control, monitorización y restricción adecuadas sobre medios extraíbles.

1. Clasificación de Requisitos

RF
  • Registro y autorización previa de dispositivos USB.
  • Monitorización del uso de puertos USB en tiempo real.
  • Generación de alertas automáticas ante cualquier conexión no autorizada.
RNF
  • Transparencia y latencia mínima en la lectura para usuarios legítimos.
  • Alta disponibilidad del agente de control en los endpoints (>99.9%).
T
  • Despliegue de módulo Device Control (bloqueo lógico de puertos).
  • Implementación de agente DLP de endpoint para prevenir extracciones.
  • Integración de logs con sistema SIEM.
L
  • Cumplimiento del Esquema Nacional de Seguridad (ENS) por ser Institución Pública.
  • RGPD (si la extracción afectó a datos personales).

2. Caso de Uso: Gestión USB Autorizado

Actor: Empleado autorizado
[1] El empleado solicita al departamento TI registrar un dispositivo USB.
[2] TI valida el dispositivo y lo registra en el sistema de control.
[3] La estación de trabajo del empleado permite el uso del USB autorizado.
[4] Todas las operaciones realizadas con el dispositivo quedan registradas y monitorizadas.
[Exc] USB no registrado: El sistema bloquea automáticamente el dispositivo.
[Exc] Actividad sospechosa: Desconexión forzada y alerta inmediata a seguridad.

3. Métodos de Validación

  • Auditorías internas periódicas sobre el uso real de puertos USB.
  • Pruebas técnicas para verificar el bloqueo efectivo de dispositivos no autorizados.
  • Revisión regular de políticas con los equipos de seguridad y cumplimiento.
  • Simulaciones de extracción para validar la detección del DLP y la emisión de alertas.

Ejercicio 2: Cifrado E2EE en WhatsApp/Signal

Criptografía / Privacidad
Enunciado:

El cifrado de extremo a extremo (E2EE) garantiza que solo el emisor y el receptor puedan leer los mensajes. Aplicaciones como WhatsApp y Signal utilizan el protocolo Signal que combina criptografía de curva elíptica, intercambio de claves X3DH, el algoritmo Double Ratchet y claves efímeras para maximizar la seguridad y la privacidad. E2EE protege eficazmente frente a intrusiones externas o internas, pero también ha generado debates. Mientras expertos y usuarios lo consideran indispensable para la privacidad, algunos gobiernos han propuesto puertas traseras o mecanismos de escaneo del lado del cliente lo que podría comprometer la seguridad del sistema.

1. Clasificación de Requisitos

RF
  • Garantizar que solo emisor y receptor descifren y lean los mensajes.
  • Renovar las claves criptográficas automáticamente en cada mensaje enviado.
  • Proveer un mecanismo visual (código QR) para verificar la identidad del contacto.
RNF
  • Transparencia para el usuario (cifrado imperceptible durante el uso normal).
  • Baja latencia en el procesamiento de mensajes desde dispositivos móviles.
T
  • Uso del Protocolo Signal (Algoritmo Double Ratchet para rotación de claves).
  • Criptografía de Curva Elíptica (ECC) e intercambio X3DH.
  • Cifrado simétrico AES para el contenido del mensaje (payload).
L
  • Cumplimiento del RGPD (Privacidad desde el Diseño y por Defecto).
  • Minimización de datos (El servidor de WhatsApp no almacena el contenido, solo metadatos).

2. Caso de Uso: Envío de Mensaje Cifrado

Actor: Emisor (Alice) y Receptor (Bob)
[1] Alice redacta un mensaje y presiona enviar en la aplicación.
[2] La App de Alice cifra el mensaje localmente usando la clave pública de Bob.
[3] El mensaje cifrado viaja por internet hacia los servidores de la aplicación.
[4] El servidor enruta el mensaje hacia el dispositivo de Bob (sin poder leerlo).
[5] La App de Bob recibe el mensaje y lo descifra localmente mostrando el texto.
[Exc] Intento de MitM: Falla la verificación de Códigos de Seguridad (Safety Numbers).
[Exc] Servidor comprometido: El atacante solo obtiene metadatos, no texto claro.

3. Métodos de Validación

  • Auditorías de código fuente sobre la biblioteca criptográfica utilizada en la App.
  • Análisis de tráfico de red (ej. Wireshark) para confirmar que el payload viaja cifrado.
  • Pruebas de Pentesting en dispositivos móviles para asegurar el aislamiento de las claves privadas (Keystore/Secure Enclave).
  • Verificación "Out-of-Band": Comprobación presencial o por otra vía de los códigos QR entre usuarios.

Ejercicio 3: IDS basado en ML en IoT Industrial

OT / SCADA / IA
Enunciado:

Una planta industrial desplegó un sistema IDS basado en técnicas de machine learning para identificar anomalías en redes IoT y sistemas SCADA. El objetivo era detectar comportamientos ofensivos que los IDS tradicionales basados en firmas no eran capaces de identificar. El sistema detectó desviaciones sutiles en sensores y PLCs como tráfico fuera de horario, cambios inesperados en patrones de comunicación o comandos anómalos. Esto permitió aislar dispositivos comprometidos antes de impactar la producción. El caso demuestra el valor del ML para detectar 0-days, así como los retos: datos representativos, modelos ajustados al contexto y personal OT/ML especializado.

1. Clasificación de Requisitos

RF
  • Analizar el comportamiento de dispositivos IoT, sensores, actuadores y PLCs en tiempo real.
  • Crear una línea base (profiling) del comportamiento normal de la planta.
  • Detectar anomalías mediante modelos estadísticos o algoritmos de Machine Learning (ML).
  • Generar alertas tempranas para el equipo de OT (Operational Technology) facilitando el aislamiento.
RNF
  • Operación 100% pasiva: no debe introducir latencia en la red de control industrial.
  • Minimizar la tasa de Falsos Positivos (FP) para no provocar paradas innecesarias de producción.
  • Alta resiliencia frente a entornos con datos ruidosos o incompletos.
T
  • Despliegue fuera de línea mediante puertos SPAN/Network TAPs.
  • Deep Packet Inspection (DPI) para protocolos industriales (Modbus TCP, DNP3, OPC UA).
  • Algoritmos ML de detección de anomalías (ej. Isolation Forest, Autoencoders).
L
  • Cumplimiento del estándar IEC 62443 (Ciberseguridad en redes y sistemas industriales).
  • Directiva europea NIS2 (por tratarse de infraestructura crítica u operador esencial).

2. Caso de Uso: Detección Anomalías IoT

Actor: Sistema IDS-ML / Operador OT
[1] El sistema IDS monitoriza el tráfico industrial pasivamente (SPAN).
[2] El motor ML perfila la línea base de operaciones (horarios, comandos habituales).
[3] El IDS detecta una desviación sutil (ej. escritura en PLC fuera de turno).
[4] Se genera una alerta de criticidad alta en el dashboard del SOC/OT.
[5] El operador OT valida la alerta y aísla lógicamente el sensor sin parar la planta.
[Exc] Datos incompletos o ruidosos: Aumentan los falsos positivos; el modelo requiere ajuste/limpieza.
[Exc] Ataques camuflados: El malware imita el tráfico normal; requiere modelos híbridos (firmas + ML).

3. Métodos de Validación

  • Entrenar y probar el modelo con datasets reales del entorno industrial, no simulados.
  • Crear escenarios de ataque controlados (Red Teaming) contra PLCs, sensores y SCADA.
  • Evaluar periódicamente métricas del modelo (Precisión, Recall, F1-Score) y monitorizar la lista de falsos positivos.
  • Validar la integración OT/IT mediante pruebas de respuesta coordinadas ante alertas.
  • Revisar la escalabilidad del modelo ante cambios operativos (cambio de turnos, estaciones, ciclos de producción).
  • Verificar que las alertas generadas sean accionables y no saturen de ruido al equipo OT.

Ejercicio 4: MitM en Wi-Fi Público

Redes Inalámbricas
Enunciado:

En 2021, investigadores demostraron lo sencillo que resulta ejecutar un ataque Man in the Middle (MITM) en redes Wi-Fi públicas que usan WEP, WPA débil o configuraciones obsoletas. Mediante un punto de acceso falso (Evil Twin) montado en una cafetería, lograron capturar credenciales e interceptar tráfico de decenas de usuarios que se conectaron a la red sin verificar su autenticidad. La demostración subrayó la persistencia de infraestructuras inalámbricas desactualizadas y la necesidad urgente de migrar a WPA3, reforzar mecanismos antifraude (detección de Rogue AP) y promover el uso de VPN en redes abiertas.

1. Clasificación de Requisitos

RF
  • Autenticación segura mediante WPA3 o WPA2-Enterprise con EAP.
  • Detección y alertado de puntos de acceso falsos (Rogue AP Detection).
  • Cifrado automático del tráfico inalámbrico desde el momento de asociación.
  • Aplicar aislamiento de clientes (Client Isolation) para evitar ataques laterales.
RNF
  • Alta escalabilidad y gestión de conexiones masivas concurrentes sin pérdida de rendimiento.
  • Roaming transparente (Fast BSS Transition) sin cortes en la conexión segura.
T
  • Protocolo WPA3-SAE (Simultaneous Authentication of Equals) contra ataques de diccionario.
  • WIDS/WIPS (Wireless Intrusion Detection/Prevention System).
  • Autenticación 802.1X con servidor RADIUS (EAP-TLS).
L
  • RGPD (Protección de los datos personales en tránsito de los usuarios conectados).
  • Términos y Condiciones / Políticas de Uso Aceptable (AUP) en portales cautivos.

2. Caso de Uso: Autenticación Segura en Wi-Fi

Actor: Usuario Móvil y Servidor RADIUS
[1] El usuario selecciona la red corporativa/segura.
[2] El Access Point (AP) bloquea el tráfico de datos y reenvía la petición al servidor RADIUS (802.1X).
[3] El servidor valida las credenciales o el certificado digital del cliente (EAP-TLS).
[4] Tras el éxito, se negocian claves de sesión únicas y el usuario accede cifrado.
[Exc] AP Falso detectado (MAC Spoofing): El WIPS bloquea el SSID fraudulento y alerta al SOC.
[Exc] Credenciales erróneas/Certificado revocado: Acceso denegado, se registra el intento.

3. Métodos de Validación

  • Pentesting inalámbrico (usando suites como Aircrack-ng o Kismet) para verificar la detección de un Evil Twin en tiempo real.
  • Auditoría de protocolos para asegurar que no se permiten "Downgrade Attacks" (forzar a WEP/WPA).
  • Comprobación activa del "Client Isolation" intentando hacer ping o ataques laterales entre dos dispositivos de la misma red pública.
  • Verificación de las políticas corporativas que obliguen al uso de VPN en redes abiertas (MDM o políticas de endpoint).

Ejercicio 5: Firewall en la Nube (AWS)

Cloud Security
Enunciado:

Una startup que operaba en AWS configuró incorrectamente sus Security Groups dejando abierto el acceso SSH (puerto 22) a internet en varias instancias críticas. Un escaneo automatizado identificó la exposición y permitió a atacantes acceder a los servidores, instalar cryptominers y robar datos de clientes. La investigación posterior reveló la inexistencia de políticas de seguridad específicas para la nube, ausencia de validaciones automáticas y falta de herramientas de Cloud Security Posture Management (CSPM). El caso ilustra cómo los errores de configuración son una de las principales causas de incidentes en entornos cloud.

1. Clasificación de Requisitos

RF
  • Definir reglas estrictas para puertos críticos (ej. SSH puerto 22, RDP, bases de datos), bloqueando su acceso público a Internet.
  • Implementar un sistema de validaciones automáticas previas a cualquier cambio en los Security Groups.
  • Establecer y aplicar políticas de seguridad específicas para el entorno de la nube.
RNF
  • Automatización máxima para minimizar el error de configuración humana.
  • Visibilidad continua y en tiempo real del estado de exposición de la infraestructura.
T
  • Herramientas de Cloud Security Posture Management (CSPM) [mencionado en el texto].
  • AWS Security Groups configurados por defecto en modo denegación.
L
  • RGPD / Leyes de Protección de Datos (debido al "robo de datos de clientes").
  • Políticas de seguridad internas específicas para la nube [mencionado en el texto].

2. Caso de Uso: Control de accesos en Security Groups

Actor: Administrador TI / Sistema de Validación
[1] El administrador intenta modificar un Security Group para abrir el puerto 22.
[2] El sistema de validaciones automáticas intercepta y analiza la regla solicitada.
[3] La validación detecta que el origen es "Internet" y viola la política de seguridad cloud.
[4] Se bloquea la configuración, evitando la exposición, y se alerta al equipo.
[Exc] Modificación manual saltando la validación: La herramienta CSPM lo detecta en su escaneo y revoca la regla.

3. Mecanismos para validar configuraciones

  • Realizar escaneos automatizados propios (simulando el ataque descrito) para verificar que no hay puertos críticos expuestos a internet.
  • Desplegar y auditar el entorno mediante la herramienta CSPM para asegurar que cumple con las políticas de seguridad en la nube establecidas.
  • Intentar aplicar intencionadamente una regla insegura en el Security Group para comprobar que las validaciones automáticas la bloquean.

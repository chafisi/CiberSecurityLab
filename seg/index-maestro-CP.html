🛡️ Caso Práctico 3: Ataque al algoritmo DES y transición hacia AES

Análisis visual de la obsolescencia criptográfica, matriz de requisitos para la transición a cifrado robusto, flujo de actualización y validación de normativas NIST/ENISA.

💡 1. Matriz de Requisitos

🟢 FUNCIONALES
  • Algoritmos oficiales (recomendados)
  • Actualización modular sin interrupciones
  • Auditoría/registro de algoritmos y claves
🔵 NO FUNC.
  • Tiempos de cifrado / Rendimiento
  • Compatibilidad temporal (Sistemas heredados)
🟣 TÉCNICOS
  • Sustituir DES por AES (128/192/256 bits)
  • Emplear librerías probadas (OpenSSL, libsodium)
  • Análisis periódico del estado criptográfico

🔄 2. Flujo de Actualización Criptográfica

Actor: Administrador de seguridad / Equipo TI
⚠️ Sistema detecta algoritmo inseguro (DES)
⬇️
🚨 Genera alerta al Administrador
⬇️
⚙️ Selección nuevo algoritmo (AES/SHA-256)
⬇️
🔄 Aplica actualización y migra datos/claves
⬇️
✅ Valida compatibilidad y documenta
⚠️ EXCEPCIONES:
❌ Incompatible (Heredado) ➔ 🛡️ Activar plan mitigación / gateway
❌ Fallo Migración ➔ 🔙 Restauración desde copia segura

📊 3. Dashboard de Validación

📋 Auditoría Periódica Verificar uso exclusivo de algoritmos vigentes
🤖 Análisis Automatizado Uso de testssl.sh y scripts internos
⚡ Test Rendimiento Asegurar que AES cumple requisitos operativos
⚖️ Compliance Revisión de normativas NIST / ENISA

💡 1. Matriz de Requisitos

🟢 FUNCIONALES
  • Validar certificados (OCSP/CRL)
  • Registrar intentos de emisión/acceso
  • Alertas ante dominios no autorizados
🔵 NO FUNC.
  • Alta disponibilidad (OCSP, CRL)
  • Baja latencia en verificación
🟣 TÉCNICOS
  • Certificate Transparency y monitorización de logs públicos
  • Endurecer CA: HSM, MFA y aislamiento estricto
  • Segmentar red para el entorno de emisión

⬇️

⬇️
🔍 Consulta estado mediante OCSP o CRL
⬇️
✅ Establece conexión cifrada (Válido)
⚠️ EXCEPCIONES:
❌ Certificado Revocado ➔ 🛑 Conexión Bloqueada
💀 CA Comprometida ➔ 🚫 Advertencia crítica y bloqueo

📊 3. Dashboard de Validación

📜 Transparency Logs Supervisión continua de registros públicos (CT)
🥷 Pentest a la PKI Pruebas de intrusión en el entorno de emisión CA
🎭 Simulación MITM Verificar la detección de certificados ilegítimos

UD 4: Cortafuegos

🔥 Caso Práctico 5: Configuración Errónea en Firewall Perimetral

Análisis visual para la prevención de errores de configuración, matriz de requisitos de seguridad, flujo de gestión de reglas (despliegue seguro) y panel de validación de políticas.

En una empresa mediana del sector financiero, una mala configuración en el cortafuegos perimetral dejó expuesto a Internet un puerto asociado a la base de datos corporativa. Aunque existían credenciales, la reutilización de contraseñas y la falta de segmentación permitieron que los atacantes obtuvieran acceso mediante fuerza bruta. El incidente provocó exfiltración de datos sensibles, interrupción de varios servicios internos y un impacto operativo significativo. La auditoría posterior reveló que no existían procedimientos formales para validar reglas del firewall, la gestión se realizaba de forma manual, no había revisión por pares, y no existía un proceso automatizado de detección de puertos expuestos.

💡 1. Matriz de Requisitos

🟢 FUNCIONALES
  • Validar reglas antes de producción
  • Mantener registro de cambios (historial)
  • Alertas por puertos no autorizados
🔵 NO FUNC.
  • Alta disponibilidad (≥99,9%)
  • Baja latencia en el filtrado
  • Interfaz clara y resistente a errores
🟣 TÉCNICOS
  • Segmentación (BD, frontend, administración)
  • Autenticación fuerte (MFA y RBAC)
  • Automatización de pruebas y uso de entorno staging
  • Principio de mínimo privilegio en políticas

🔄 2. Flujo de Gestión Segura de Reglas

Actor: Administrador de seguridad
✍️ Propone nueva regla de firewall
⬇️
👥 Revisión por pares (Peer Review)
⬇️
⚙️ Validación automatizada (Análisis)
⬇️
🧪 Aplicación en entorno de pruebas (Staging)
⬇️
🚀 Despliegue en producción y Registro
⚠️ EXCEPCIONES:
❌ Regla Insegura/Redundante ➔ 🛑 Rechazo y justificación
💥 Fallo de Rendimiento ➔ 🔙 Rollback automático

📊 3. Dashboard de Validación

📋 Auditoría Periódica Revisión de reglas y políticas de firewall
🔍 Escaneos de Red Nmap, Nessus para detectar puertos expuestos
🥷 Pentest Interno Verificar la segmentación de la red real
⚖️ Config Drift Comparar arquitectura prevista vs configuración real

💡 1. Matriz de Requisitos

🟢 FUNCIONALES
  • Inspección profunda de paquetes (DPI)
  • Identificar apps (cifradas/evasivas)
  • Bloqueo automático según políticas
  • Reportes detallados de actividad sospechosa
🔵 NO FUNC.
  • Rendimiento DPI sin latencia crítica
  • Escalabilidad para picos de tráfico
  • Alertas claras y bajos Falsos Positivos
  • Interfaz de gestión intuitiva
🟣 TÉCNICOS
  • Integración SIEM y automatización de despliegue
  • Control granular (usuario, app, categoría, contenido)
  • Inspección TLS/SSL en escenarios permitidos
  • Actualizaciones frecuentes (firmas, heurística)

🔄 2. Flujo de Detección (App no autorizada)

⬇️
🚨 Genera alerta al SOC
⬇️
📝 Registro para análisis y auditoría
⚠️ EXCEPCIONES:
❌ App legítima mal clasificada ➔ 🛡️ Revisión de reglas / Whitelisting
🔄 Falso Positivo Recurrente ➔ ⚙️ Reentrenamiento de firmas

⚡ Test Rendimiento Verificar latencia de DPI en servicios críticos
🎯 Tuning SOC Ajuste de sensibilidad para reducir falsos positivos

🎓 Caso Práctico 7: Uso de Snort en una Universidad

Análisis visual del despliegue de un NIDS (Network Intrusion Detection System), matriz de requisitos, flujo de detección de escaneos y panel de validación de reglas.

💡 1. Matriz de Requisitos

🟢 FUNCIONALES
  • Monitorizar tráfico entrante/saliente en tiempo real
  • Detectar patrones basados en firmas (escaneos, exploits)
  • Generar alertas automáticas para el SOC
  • Registrar eventos para análisis posterior
🔵 NO FUNC.
  • Analizar sin pérdida de paquetes
  • Baja tasa de falsos positivos
  • Disponibilidad continua del sensor
🟣 TÉCNICOS
  • Uso de red espejo (SPAN/TAP)
  • Integración con cortafuegos perimetral
  • Configuración de preprocessors

🔄 2. Flujo Detección (Escaneo de Puertos)

Actor: IDS Snort
👁️ Snort inspecciona tráfico en el perímetro
⬇️
🔍 Detecta patrón: Múltiples SYN consecutivos
⬇️
🚨 Genera alerta con IP origen/destino
⬇️
📊 Equipo TI revisa alerta en SIEM/Dashboard
⬇️
🛑 Firewall bloquea origen (Auto/Manual)
⚠️ EXCEPCIONES:
❌ Actividad Legítima (Inventariado) ➔ 🛡️ Ajuste reglas / Whitelist

📡 Test Inyección Uso de Nmap/Masscan para verificar alertas
⚡ Rendimiento NIDS Análisis de pérdida de paquetes y carga CPU
🧩 Correlación SIEM Cruce de logs (Syslog + Firewall + Snort)

Mapa Mental y Conceptos (Teoría)

Fundamentos de Compresión y Shannon

El objetivo de la codificación de la fuente es representar mensajes de forma eficiente eliminando redundancias. La métrica fundamental es la Entropía ($H(X)$), que establece el límite teórico de compresión.

$$H(X)=-\sum_{i=1}^{n}p(x_{i})\log_{2}p(x_{i})$$
  • Baja entropía: Alta redundancia, muy compresible (logs, texto, imágenes).
  • Alta entropía: Datos aleatorios, incompresibles (claves criptográficas).

Sin Pérdida (Lossless)

Recupera exactamente el original bit a bit.

  • Huffman: Códigos cortos a símbolos frecuentes. Óptimo para fuentes estáticas.
  • Aritmética: Codifica secuencias como subintervalos [0,1). Se acerca más a $H(X)$.
  • Lempel-Ziv (LZ77/LZ78): Aprende patrones con ventanas deslizantes (ZIP, gzip).

Con Pérdida (Lossy)

Sacrifica información no esencial para mayor ratio de compresión.

  • JPEG: Usa DCT, cuantización y luego Huffman
  • MP3 / MPEG: Para audio y vídeo.

Pipeline Seguro: CtE (Compress-then-Encrypt)

Comprimir elimina patrones; cifrar oculta la estructura. Orden correcto para evitar fugas:

1. Compresión
➔
2. Cifrado (AEAD)
➔
3. Transporte (TLS)

Laboratorios de Técnicas

Lab 1: Algoritmo de Huffman

Asigna bits más cortos a las letras más probables.

A (50%) → 0 | B (25%) → 10 | C (25%) → 11

Lab 2: Lempel-Ziv (LZ77)

Utiliza una ventana deslizante para buscar repeticiones y genera tripletas (desplazamiento, longitud, literal).

A B A B A B A A B

ABA BABAAB
ABA BABA AB → Puntero: (offset=2, length=4)

Lab 3: Codificación Aritmética

Refina un intervalo [0, 1) iterativamente. Es más eficiente que Huffman pero computacionalmente más compleja.

A [0, 0.5)
B [0.5, 0.75)
C [0.75, 1)

Secuencia 'A' reduce el intervalo al primer bloque. Siguiente letra refina dentro de [0, 0.5).

3

Perspectiva Auditoría: Triada CID

C Confidencialidad

La compresión no aporta confidencialidad. Si se co-comprimen secretos (cookies) junto con datos controlados por el atacante, la reducción del tamaño del ciphertext revela el secreto.

Ataques CRIME y BREACH: Explotan la compresión TLS y HTTP para crear oráculos de longitud. Auditoría: Desactivar compresión TLS, usar padding aleatorio y separar contextos de compresión.

I Integridad

La codificación de fuente asume que el canal es fiable. Una alteración de 1 bit en un flujo Huffman o LZ desincroniza todo el decodificador.

Auditoría AEAD: Es obligatorio el uso de cifrado autenticado (AES-GCM, ChaCha20-Poly1305) posterior a la compresión para garantizar la integridad.

D Disponibilidad

Un algoritmo de compresión mal dimensionado consume excesiva CPU y RAM, abriendo vectores de Denegación de Servicio (DoS).

Zip-Bombs y Resource Exhaustion: Atacantes envían archivos altamente comprimidos que, al descomprimirse, agotan la memoria. Auditoría: Imponer límites de tamaño, timeouts y controlar profundidad de recursión.

Tarjetas de Memorización Flash

Entropía $H(X)$

Incertidumbre media y límite teórico de compresión. Alta entropía = claves, incompresible.

Pipeline CtE

Compress-then-Encrypt. Comprimir antes de cifrar elimina patrones y reduce carga.

CRIME / BREACH

Ataques que explotan co-compresión de secretos e inputs para inferir la cookie midiendo el tamaño del ciphertext.

LZ77 vs Huffman

Huffman requiere conocer probabilidades. LZ77 es universal, aprende patrones al vuelo usando ventana deslizante.

Zip-bomb

Ataque de DoS donde un archivo diminuto se expande gigabytes al descomprimirse, saturando la memoria.

Padding

Mitigación técnica para CRIME/BREACH. Consiste en rellenar con datos aleatorios para destruir el oráculo de longitud.

Caso 15: Huffman en JPEG

Enunciado

Imágenes subidas se comprimen (JPEG/Huffman) acelerando transmisión. Se cifran (TLS 1.3), pero la longitud del ciphertext revela información indirecta (resolución, tipo) creando canal lateral.

1. Requisitos Seguros

  • F: Comprimir antes de cifrar; aplicar padding opcional; usar TLS 1.3.
  • NF: Minimizar latencia, reducir tamaño significativamente, evitar patrones correlacionables.
  • T: JPEG/Huffman, AES-GCM, eliminar EXIF, padding uniforme.

2. Caso de Uso

  • Usuario sube imagen.
  • Aplicación convierte a JPEG (Huffman).
  • Comunicación se cifra en TLS 1.3.
  • Servidor procesa y registra tamaño.
  • Excepción: Aplicar padding si el tamaño revela info sensible.

3. Validación

  • Pruebas canal lateral por longitud.
  • Evaluación rendimiento (ahorro vs PNG/RAW).
  • Verificación TLS (SSL Labs).
  • Comprobar eliminación metadatos EXIF.

💡 Correlación Ignacio

"archivos más pequeños" → Requisito Funcional: Comprimir antes.
"longitud revela información" → Req Téc: Padding uniforme y Validación: Pruebas de canal lateral por longitud.
"filtrado de metadatos" → Req Téc: Eliminación metadatos EXIF.

C16

Caso 16: CRIME y Filtración (TLS/HTTP)

Enunciado

Ataque CRIME infiere cookies co-comprimiendo datos del atacante y secretos. El tamaño del ciphertext revela coincidencias. Relevante en HPACK/QPACK (HTTP/2-3).

1. Requisitos Seguros

  • F: Separar datos secretos de inputs controlados; desactivar compresión TLS.
  • NF: Mantener rendimiento sin TLS comp; minimizar latencia por mitigaciones.
  • T: Mitigaciones BREACH (Padding aleatorio, Rate limit, Fragmentación); configurar HPACK seguro.

2. Caso de Uso

  • Cliente envía parámetros controlados.
  • Servidor genera respuesta con cookie.
  • Sistema evita co-comprimir: usa HPACK separado o padding.
  • Cifra TLS 1.3 (sin compresión protocolo).
  • Excepción: Separación estricta si HTTP/2 no tiene políticas seguras.

3. Validación

  • Entorno pruebas especializado (CRIME/BREACH).
  • Análisis de tráfico cifrado (validar TLS comp off).
  • Revisión políticas HPACK/QPACK (cookies aisladas).
  • Pruebas de padding aleatorio.

💡 Correlación Ignacio

"explota la compresión a nivel TLS" → Req Téc: Desactivar compresión TLS.
"co-comprime datos" → Req Func: Separar datos secretos.
"HPACK/QPACK pueden crear canales" → Req Téc: Configuración segura HPACK.

C15

Caso 15: Huffman en JPEG y Transmisión Segura

Enunciado del Caso

Muchas plataformas procesan imágenes subidas convirtiéndolas a JPEG (transformación DCT, cuantización y Huffman). Aunque se transmiten por HTTPS/TLS 1.3, la longitud del ciphertext puede revelar información indirecta (resolución, tipo). Esto crea un canal lateral si no se normaliza el tamaño o aplica padding. El caso analiza la interacción entre compresión y cifrado.

1. Requisitos para un pipeline seguro "comprimir → cifrar"

Funcionales:
  • Comprimir imágenes antes del cifrado.
  • Transmitir mediante canal cifrado (TLS 1.3).
  • Aplicar padding o normalización si es necesario.
No funcionales:
  • Minimizar latencia en conversión.
  • Reducir tamaño significativamente.
  • Evitar patrones correlacionables en tamaño.
Técnicos:
  • Codificación Huffman (JPEG).
  • Algoritmos robustos (AES-GCM).
  • Eliminar metadatos EXIF.

2. Elaborar un caso de uso

Caso de uso: Pipeline seguro "comprimir → cifrar" en el envío de imágenes.

Actor: Usuario y Servidor seguro.

Flujo principal:
  1. El usuario selecciona y sube una imagen al sistema.
  2. La aplicación convierte la imagen a JPEG usando codificación Huffman.
  3. La comunicación con el servidor se cifra mediante TLS 1.3.
  4. El servidor recibe la imagen comprimida y la procesa o almacena.
  5. El sistema registra el tamaño resultante para análisis interno o estadísticas.
Excepciones:
  • La imagen contiene metadatos sensibles → se realiza una normalización previa.
  • El tamaño del archivo podría revelar información → se aplica padding uniforme antes del cifrado.
  • Fallo en la compresión → se reintenta con parámetros conservadores.

3. Métodos de validación frente a oráculos de longitud

  • Pruebas de canal lateral por longitud: comparar archivos similares para verificar si la longitud del ciphertext permite inferencias (resolución, detalle).
  • Evaluación de rendimiento: comparar el ahorro de ancho de banda frente a PNG o RAW.
  • Verificación TLS: uso de SSL Labs o Wireshark para confirmar cifrado correcto sin fallback.
  • Análisis de privacidad: comprobar eliminación de EXIF.
  • Pruebas con padding: medir si el padding anula la deducibilidad de tamaños.

C16

Caso 16: CRIME en TLS/HTTP y Filtración de Cabeceras

Enunciado del Caso

CRIME explota la compresión a nivel TLS/SPDY para inferir secretos (cookies). Si un servidor co-comprime datos del atacante y secretos, la variación de longitud del ciphertext revela coincidencias. Relevante en HTTP/2-3 (HPACK/QPACK), creando canales laterales si no se gestionan correctamente los contextos de compresión.

1. Requisitos para evitar filtraciones por compresión

Funcionales:
  • Separar datos secretos de entradas del usuario.
  • Desactivar compresión TLS si no aporta beneficio.
  • Gestionar contextos (HTTP/2 y 3).
No funcionales:
  • Mantener rendimiento pese a desactivar TLS comp.
  • Minimizar latencia de las mitigaciones.
Técnicos:
  • Mitigaciones BREACH (Padding, Rate limit).
  • Configurar HPACK/QPACK (cabeceras aisladas).

2. Elaborar un caso de uso

Caso de uso: Evitar la co-compresión de datos sensibles.

Actor: Servidor Web o Balanceador de carga.

Flujo principal:
  1. El cliente envía petición con parámetros controlados.
  2. El servidor genera la respuesta incluyendo cookies secretas.
  3. El sistema evita co-comprimir datos sensibles (separa contextos HPACK o añade padding).
  4. La comunicación se cifra en TLS 1.3 sin compresión a nivel de protocolo.
  5. El usuario recibe respuesta sin revelar información por longitud.
Excepciones:
  • Uso de HTTP/2 o 3 sin políticas seguras → se activa separación estricta de contextos.
  • Compresión en grandes volúmenes de tráfico → Padding aleatorio obligatorio.

3. Métodos de validación de oráculos de longitud

  • Entorno de pruebas especializado (CRIME/BREACH): Comprobar variaciones de longitud al modificar parámetros.
  • Escaneos automáticos: Auditoría web para dependencias HPACK/QPACK.
  • Análisis de tráfico cifrado: Verificar que TLS no usa compresión y no hay correlación input/tamaño.
  • Revisión de políticas HPACK: Confirmar aislamiento de cookies.
  • Pruebas de padding: Evaluar si el relleno elimina el oráculo matemático.

Glosario de Términos (U8)

AEAD

Authenticated Encryption with Associated Data. Cifrado que asegura confidencialidad e integridad (ej. AES-GCM).

BREACH

Ataque que explota compresión HTTP del cuerpo de la respuesta para inferir secretos.

CRIME

Ataque que explota compresión TLS o cabeceras SPDY/HTTP inferir cookies.

DEFLATE

Algoritmo que combina LZ77 (patrones) y Huffman. Usado en ZIP, gzip y PNG.

Entropía de Shannon

Incertidumbre matemática de una fuente. Define el límite teórico al que se puede comprimir sin pérdida.

HPACK / QPACK

Sistemas de compresión de cabeceras en HTTP/2 y HTTP/3. Mal configurados causan oráculos de longitud.

Padding Aleatorio

Relleno de bytes basura añadido antes de cifrar para destruir variaciones de longitud explotables por atacantes.

Zip-bomb

Ataque físico (DoS) donde un archivo malicioso agota la CPU/RAM durante la fase de descompresión.

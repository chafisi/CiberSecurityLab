1. Amenazas y Defensa por Capas (OSI)

La seguridad no es un bloque monolítico; una medida perfecta en Capa 7 (HTTPS) es inútil si la Capa 2 (ARP) está comprometida. Se requiere Defensa en Profundidad.

L1 / L2
Física/Enlace

Fundamentos Físicos y de Red Local

Protección del medio (cables, Wi-Fi) y de la resolución de direcciones locales (MAC).

Amenaza: ARP Spoofing, Interceptación Wi-Fi (WEP). Defensa: WPA3, Port Security, Segmentación VLAN.
L3 / L4
Red/Transp

Enrutamiento y Conexión

Cifrado de túneles y garantía de entrega de paquetes (IP / TCP / UDP).

Capa 3: IP Spoofing, Routing Attacks. Defensa L3: IPSec, ACLs, Cortafuegos.
Capa 4: SYN Flood (DoS). Defensa L4: TLS, SYN Cookies, Rate Limiting.
L5 - L7
Ses/App

Sesión, Presentación y Aplicación

Gestión de la identidad del usuario, formatos de cifrado y consumo de servicios (HTTP, DNS).

  • Amenazas L5/L6: Downgrade Attacks (forzar criptografía débil), Session Hijacking.
  • Amenazas L7: Inyecciones (XSS, SQLi), manipulación de DNS.
  • Defensas: Cookies (Secure, HttpOnly), SMTPS/DNSSEC, WAF, Proxies Inversos.

2. Zero Trust, Sesiones y Legacy

⚠️ El Peligro del "Legacy" (Downgrade)

Ignacio subraya: "La compatibilidad es a veces sinónimo de riesgo". Si un servidor soporta TLS 1.3 pero mantiene activo SSL 3.0 por compatibilidad con equipos viejos, un atacante MitM puede forzar la comunicación hacia el protocolo obsoleto para romper el cifrado.

Mitigación: Deshabilitar protocolos obsoletos. "Si se mantiene legado, que sea consciente, documentado y revisado".

🍪 Gestión de Sesiones

"Muchas brechas parecen fallos de criptografía, pero son robos de sesión". Aunque uses TLS (Capa 4), si roban la cookie (Capa 7), el atacante es el usuario.

  • Session Hijacking: Robo del token. Prevención: Expiración corta y rotación.
  • Fijación de Sesión: Atacante fuerza un ID. Prevención: Regenerar ID tras Login.
  • Protección de Cookies: Banderas Secure (solo HTTPS) y HttpOnly (no accesibles vía JavaScript / evita XSS).

🛡️ El Enfoque Zero Trust (Perímetro Difuso)

En redes modernas (Teletrabajo, Cloud), el firewall perimetral no basta. "Nunca confíes, verifica siempre".

1. Identidad: MFA robusto, no fiarse de la IP origen.
2. Segmentación: Micro-perímetros por servicio.
3. Verificación: Análisis continuo del estado del dispositivo.

3. Auditoría: Limitaciones y Retos

🙈

La Paradoja del Cifrado (Visibilidad vs. Confidencialidad)

Implementar cifrado masivo (TLS/IPSec) protege los datos contra escuchas (Capa Física/Enlace), pero ciega a los sistemas de defensa propios (IDS/IPS/Firewalls). El auditor debe evaluar el riesgo de no ver el payload malicioso, justificando la necesidad de inspección SSL (SSL Decryption) mediante proxies autorizados.

⚖️

Seguridad vs. Rendimiento y Disponibilidad

Escalar la seguridad impone límites. Inspeccionar profundamente (DPI) o cifrar todo el tráfico de Capa 3 añade latencia. Un exceso de control puede provocar la caída de servicios críticos (Falso positivo en IPS o cuellos de botella). El auditor exige SLA de red y pruebas de estrés.

🧩

Interoperabilidad

Conectar sistemas heterogéneos (ej. VPNs IPSec entre sedes con equipos de distinto fabricante) genera fricciones. Una mala negociación de parámetros criptográficos durante el Handshake abre la puerta a ataques o caídas de conexión.

4. Fichas Rápidas (Memorización)

Downgrade Attack

CAPA 5/6

El atacante intercepta la negociación de seguridad e interfiere para forzar al servidor y cliente a usar un protocolo obsoleto y vulnerable (ej. de TLS 1.3 a SSL 3.0).

ARP Spoofing

CAPA 2

Falsificación de mensajes ARP en la red local. El atacante asocia su dirección MAC con la IP del Gateway, redirigiendo todo el tráfico LAN hacia su equipo (Man-in-the-Middle).

XSS (Cross-Site Scripting)

CAPA 7

Inyección de código malicioso (JS) en páginas web legítimas. Sirve para secuestrar sesiones robando cookies que no tienen el flag "HttpOnly".

SYN Flood

CAPA 4

Ataque DoS. El atacante envía miles de peticiones TCP SYN sin completar el handshake, agotando la memoria del servidor (estado de conexiones semi-abiertas).

6. Glosario Técnico (Capas E2E)

IPSec

Suite de protocolos (Capa 3) que autentica y cifra paquetes IP. Core de las VPNs corporativas entre sedes.

TLS

Transport Layer Security (Capa 4/5). Cifra el canal de comunicación. Sucesor moderno de SSL.

WPA3

Estándar de seguridad Wi-Fi (Capa 1/2). Mitiga ataques de diccionario comunes en WPA2 mediante cifrado individualizado.

HttpOnly / Secure

Banderas de seguridad en cookies (Capa 7). Secure exige HTTPS; HttpOnly impide acceso vía scripts locales (XSS).

Zero Trust

Arquitectura que elimina la confianza implícita del perímetro. Exige verificación continua de identidad y postura del dispositivo.

CSP

Content Security Policy. Cabecera HTTP que previene XSS limitando desde qué dominios puede cargar scripts un navegador.

Flujo Operativo: Arquitectura Zero Trust

Cómo se defiende en profundidad una petición desde el cable físico hasta la base de datos de la aplicación.

L1
Acceso Físico y Wi-Fi (802.1X / WPA3)

El dispositivo se conecta. No hay confianza automática. Se autentica el puerto o la conexión inalámbrica mediante certificados.

L3
Microsegmentación (SDN / Firewalls)

La red aísla al usuario. Solo puede ver (routing) el servidor web específico que necesita, el resto de la red está invisible e inaccesible.

L4
Túnel Cifrado Fuerte (TLS 1.3)

Se establece el canal. El servidor rechaza cualquier intento de Downgrade a TLS 1.2 o SSL, garantizando la integridad en el transporte.

L7
Identidad, Sesión y WAF

MFA verificado. Se emite una cookie de sesión con HttpOnly. El WAF inspecciona los inputs para bloquear XSS/SQLi antes de llegar al código del backend.
